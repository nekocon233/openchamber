# Desktop Tunnels

The desktop app can create a public tunnel from **Settings → OpenChamber → Tunnel**. You do not need to start OpenChamber from the CLI for this path.

## Install a provider

OpenChamber starts the provider CLI on your machine. Install the provider you want to use first:

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare uses `cloudflared`. Ngrok requires an ngrok account and an authtoken from the ngrok dashboard:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC requires no manual client installation. OpenChamber downloads and verifies its pinned FRPC 0.70.0 binary automatically.

## Start from the app

1. Open **Settings → OpenChamber → Tunnel**.
2. Choose **Cloudflare**, **FRPC**, or **Ngrok**.
3. Start a supported tunnel mode.
4. Scan the generated QR code from your phone.

Ngrok currently supports quick tunnels. Cloudflare supports quick tunnels and managed Cloudflare modes. FRPC supports managed-remote TCP and HTTP-vhost endpoints through your own FRPS server.

## Configure FRPC

Settings offers **TCP** and **HTTP vhost** as the FRPC proxy type. Enter the FRPS address, control port, and shared token in the private token field; there is no CA field. OpenChamber supplies its local port automatically for either type. OpenChamber builds a private temporary PEM from the host runtime trust roots and injects it into FRPC, which still enables `transport.tls` and verifies the FRPS certificate against a `serverName` equal to the FRPS address. OpenChamber never learns a CA from an unauthenticated FRPS and never disables certificate verification. If FRPS uses a private or enterprise CA, install that CA into the OS trust store of the machine running OpenChamber (or expose it through `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`) instead of configuring it per tunnel. For compatibility, CA bundles at the legacy host paths `~/.config/frp/ca.crt` and `~/.frp/ca.crt` (if present) are also merged into the host trust roots; there is no per-tunnel CA field in Settings.

- **TCP:** enter the remote mapping port and the exact external public HTTPS origin. FRPC forwards plain HTTP on the mapped port, so keep that FRPS port loopback-only and place Caddy or another TLS terminator at the declared URL. OpenChamber rejects HTTP, malformed, or non-origin URLs and never infers browser TLS from the FRPS control address.
- **HTTP vhost:** enter the FRPS routing host (custom domain) and the external public hostname. The public URL is `https://<hostname>`. Caddy terminates HTTPS on `443`, forwards to the FRPS vhost HTTP port, sets `Host` to the custom domain, and sets `X-Forwarded-Host` to the public hostname.

A shared FRPS token authenticates FRPC clients; it does not provide isolation between HTTP vhosts.

## Access protection

Even when the provider URL itself is public, OpenChamber still protects access with its own connect token. The generated connect link contains a one-time token, has a TTL, and old unused links are revoked when you generate a new one or stop/restart the tunnel.
