# Túneles

Un túnel es un enlace público a tu OpenChamber, para que puedas acceder a él desde un navegador corriente en otra red. Usa `openchamber tunnel` para crear uno para una instancia en marcha.

> Conectar tus **propios dispositivos** (la app móvil, otro escritorio) normalmente no necesita un túnel: [vincúlalos](/es/connect-devices/) y deja que el [Private Relay](/es/private-relay/), cifrado de extremo a extremo, se encargue del acceso fuera de casa sin configuración.

## Requisitos previos

OpenChamber inicia el CLI del proveedor de túnel en tu máquina. Para Cloudflare o Ngrok, instala primero el proveedor que quieras usar:

```bash
brew install cloudflared
brew install ngrok
```

Los túneles rápidos de Cloudflare pueden funcionar con `cloudflared`. Ngrok requiere una cuenta de ngrok y un authtoken del panel de ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC no requiere instalación manual. OpenChamber descarga y gestiona automáticamente FRPC 0.70.0.

## Inicio rápido

1. Inicia OpenChamber:

```bash
openchamber
```

Si omites este paso, `openchamber tunnel start` puede iniciar automáticamente un servidor CLI. Al hacerlo, puedes pasar opciones del servidor como `--port`, `--host`, `--lan`, `--ui-password` y `--api-only`.

2. Inicia un túnel de Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

O inicia un túnel de Ngrok:

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. Comprueba el estado:

```bash
openchamber tunnel status
```

Cuando el túnel está activo, `status` muestra una URL pública. Ábrela, o escanea el código QR, para acceder a OpenChamber desde cualquier lugar.

Por defecto, OpenChamber imprime un código QR en sesiones TTY interactivas. Usa `--qr` para forzar la salida del QR, o `--no-qr` para desactivarla.

## Proveedores

- `cloudflare`: modos quick, managed remote y managed local
- `ngrok`: modo quick
- `frpc`: modo managed remote con endpoint TCP o host virtual HTTP mediante tu propio servidor FRPS

## Modos gestionados

### Gestionado remoto

Usa un token + nombre de host gestionados por Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### Gestionado local

Usa una configuración local de `cloudflared`:

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC (managed-remote)

FRPC conecta el puerto local activo de OpenChamber con tu propio servidor FRPS. OpenChamber proporciona automáticamente su puerto de escucha local para ambos tipos de endpoint.

No se requiere ninguna configuración de CA. OpenChamber reúne las raíces de confianza del entorno de ejecución en el que corre en un PEM temporal privado y lo inyecta en FRPC, que sigue activando `transport.tls` y verifica el certificado de FRPS con un `serverName` igual a `--frps-address`. OpenChamber nunca aprende una CA de un FRPS no autenticado y nunca desactiva la verificación del certificado. Si tu FRPS usa una CA privada o empresarial, instala esa CA en el almacén de confianza del sistema operativo de la máquina que ejecuta OpenChamber (o expónla mediante `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`); ya no existe un campo de CA por túnel. Por compatibilidad, los paquetes de CA en las rutas heredadas del host `~/.config/frp/ca.crt` y `~/.frp/ca.crt` (si existen) también se fusionan en estas raíces de confianza. En Docker, las raíces de confianza del «host» son el almacén de confianza del contenedor, así que instala o monta la CA dentro del contenedor.

#### Asignación TCP

Usa `--remote-port` para crear una asignación TCP y declara el origen HTTPS externo exacto con `--public-url`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC reenvía HTTP sin cifrar mediante la asignación TCP, por lo que OpenChamber nunca deduce una URL de navegador a partir de la dirección FRPS. `--public-url` debe ser una URL de origen `https://` donde Caddy u otro terminador TLS escuche y reenvíe al puerto loopback de FRPS. Las URL HTTP, con credenciales, rutas, consultas o mal formadas se rechazan antes de emitir un token de conexión.

#### Host virtual HTTP

Usa `--custom-domain <frps-routing-host>` junto con `--hostname <external-public-host>`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC genera un proxy `type = "http"`. Caddy termina el HTTPS público en el puerto `443` y reenvía al puerto HTTP de hosts virtuales de FRPS. Debe enviar el custom domain como `Host` para que FRPS seleccione el proxy, y el hostname externo como `X-Forwarded-Host`. Por ejemplo, si el puerto HTTP de hosts virtuales de FRPS es `8080`:

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

La URL pública tiene el formato `https://<hostname>`; en este ejemplo es `https://app.example.com`. Un token FRPS compartido autentica a los clientes FRPC, pero no aísla este host virtual HTTP de otros clientes que usen el mismo token.

Con `openchamber tunnel`, FRPC solo acepta tokens mediante `--token-file`, `--token-stdin` o la solicitud interactiva privada; `--token` se rechaza. Los perfiles creados mediante una de esas entradas se pueden reutilizar.

## Perfiles (managed-remote)

Guarda un perfil reutilizable:

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

Para FRPC TCP, guarda también los datos del servidor y del puerto remoto:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

Para el host virtual HTTP de FRPC:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

Inicia usando cualquiera de los perfiles guardados:

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## Entorno para el inicio directo

El inicio directo usa los ajustes comunes del servidor FRPC y el token. Elige exactamente un endpoint: `OPENCHAMBER_TUNNEL_REMOTE_PORT` junto con `OPENCHAMBER_TUNNEL_PUBLIC_URL` para TCP, o `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN` y `OPENCHAMBER_TUNNEL_HOSTNAME` juntos para el host virtual HTTP. La verificación de FRPS no necesita ninguna variable de entorno: OpenChamber construye el ancla de confianza a partir de las raíces de confianza del entorno de ejecución anfitrión (en Docker, el almacén de confianza del contenedor), más los paquetes de CA heredados en `~/.config/frp/ca.crt` y `~/.frp/ca.crt` cuando existen.

**TCP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**Host virtual HTTP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

El puerto local de OpenChamber sigue siendo automático.

## Comandos útiles

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## Notas de comportamiento

- un único túnel activo por instancia de OpenChamber (puerto)
- iniciar un nuevo modo/proveedor en la misma instancia reemplaza el túnel anterior
- generar un nuevo enlace de conexión revoca el anterior sin usar
- el autoarranque del túnel conserva flags del servidor como `--ui-password` y `--api-only` en los ajustes de instancia usados por reinicios y actualizaciones
