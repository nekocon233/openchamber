# Tunnels

Un tunnel est un lien public vers votre OpenChamber, pour y accéder depuis un navigateur ordinaire sur un autre réseau. Utilisez `openchamber tunnel` pour en créer un pour une instance en cours d’exécution.

> Connecter vos **propres appareils** (l’application mobile, un autre desktop) ne nécessite généralement pas de tunnel — [associez-les](/connect-devices/) plutôt et laissez le [Relais privé](/private-relay/), chiffré de bout en bout, gérer l’accès hors du domicile sans aucune configuration.

## Prérequis

OpenChamber démarre la CLI du fournisseur de tunnel sur votre machine. Pour Cloudflare ou Ngrok, installez d’abord le fournisseur que vous voulez utiliser :

```bash
brew install cloudflared
brew install ngrok
```

Les tunnels rapides Cloudflare peuvent fonctionner avec `cloudflared`. Ngrok nécessite un compte ngrok et un authtoken depuis le dashboard ngrok :

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC ne nécessite aucune installation manuelle. OpenChamber télécharge et gère automatiquement FRPC 0.70.0.

## Démarrage rapide

1. Démarrez OpenChamber :

```bash
openchamber
```

Si vous sautez cette étape, `openchamber tunnel start` peut démarrer automatiquement un serveur CLI. Lors de l’auto-démarrage, vous pouvez passer des options serveur comme `--port`, `--host`, `--lan`, `--ui-password` et `--api-only`.

2. Démarrez un tunnel Cloudflare :

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

Ou démarrez un tunnel Ngrok :

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. Vérifiez l’état :

```bash
openchamber tunnel status
```

Quand le tunnel est actif, `status` affiche une URL publique. Ouvrez-la, ou scannez le QR code, pour joindre OpenChamber depuis n’importe où.

Par défaut, OpenChamber affiche un QR code dans les sessions TTY interactives. Utilisez `--qr` pour forcer la sortie QR, ou `--no-qr` pour la désactiver.

## Fournisseurs

- `cloudflare` : modes quick, managed remote et managed local
- `ngrok` : mode quick
- `frpc` : mode managed remote avec endpoint TCP ou hôte virtuel HTTP via votre propre serveur FRPS

## Modes gérés

### Managed remote

Utilisez un token + hostname gérés par Cloudflare :

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### Managed local

Utilisez une configuration `cloudflared` locale :

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC (managed-remote)

FRPC connecte le port local actif d’OpenChamber à votre propre serveur FRPS. OpenChamber fournit automatiquement son port d’écoute local pour les deux types d’endpoint.

Aucune configuration d’AC n’est requise. OpenChamber rassemble les racines de confiance de l’environnement d’exécution dans lequel il tourne dans un PEM temporaire privé et l’injecte dans FRPC, qui active toujours `transport.tls` et vérifie le certificat de FRPS avec un `serverName` égal à `--frps-address`. OpenChamber n’apprend jamais une AC auprès d’un FRPS non authentifié et ne désactive jamais la vérification du certificat. Si votre FRPS utilise une AC privée ou d’entreprise, installez cette AC dans le magasin de confiance du système d’exploitation de la machine qui exécute OpenChamber (ou exposez-la via `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`) ; il n’existe plus de champ d’AC par tunnel. Par compatibilité, les bundles d’AC présents aux chemins hérités de l’hôte `~/.config/frp/ca.crt` et `~/.frp/ca.crt` (s’ils existent) sont également fusionnés dans ces racines de confiance. Sous Docker, les racines de confiance de « l’hôte » sont le magasin de confiance du conteneur : installez ou montez l’AC dans le conteneur.

#### Mappage TCP

Utilisez `--remote-port` pour créer un mappage TCP et déclarez l’origine HTTPS externe exacte avec `--public-url` :

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC transmet du HTTP non chiffré via le mappage TCP ; OpenChamber ne déduit donc jamais une URL de navigateur de l’adresse FRPS. `--public-url` doit être une origine `https://` sur laquelle Caddy ou un autre terminateur TLS écoute et transmet vers le port FRPS local. Les URL HTTP, avec identifiants, chemin, requête ou mal formées sont rejetées avant l’émission d’un token de connexion.

#### Hôte virtuel HTTP

Utilisez `--custom-domain <frps-routing-host>` avec `--hostname <external-public-host>` :

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC génère un proxy `type = "http"`. Caddy termine le HTTPS public sur le port `443` et transmet au port HTTP des hôtes virtuels FRPS. Il doit définir `Host` sur le custom domain pour que FRPS choisisse le proxy, et `X-Forwarded-Host` sur le hostname externe. Par exemple, si le port HTTP des hôtes virtuels FRPS est `8080` :

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

L’URL publique suit le format `https://<hostname>` ; dans cet exemple, elle est `https://app.example.com`. Un token FRPS partagé authentifie les clients FRPC, mais n’isole pas cet hôte virtuel HTTP des autres clients qui utilisent ce token.

Avec `openchamber tunnel`, FRPC accepte les tokens uniquement via `--token-file`, `--token-stdin` ou l’invite interactive privée ; `--token` est refusé. Les profils créés avec l’une de ces entrées peuvent être réutilisés.

## Profils (managed-remote)

Enregistrez un profil réutilisable :

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

Pour FRPC TCP, enregistrez également les paramètres du serveur et du port distant :

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

Pour l’hôte virtuel HTTP FRPC :

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

Démarrez avec l’un des profils enregistrés :

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## Environnement de démarrage direct

Le démarrage direct utilise les paramètres communs du serveur FRPC et le token. Choisissez exactement un endpoint : `OPENCHAMBER_TUNNEL_REMOTE_PORT` avec `OPENCHAMBER_TUNNEL_PUBLIC_URL` pour TCP, ou `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN` et `OPENCHAMBER_TUNNEL_HOSTNAME` ensemble pour l’hôte virtuel HTTP. La vérification de FRPS ne requiert aucune variable d’environnement : OpenChamber construit l’ancre de confiance à partir des racines de confiance de l’environnement d’exécution hôte (sous Docker, le magasin de confiance du conteneur), ainsi que des bundles d’AC hérités situés aux chemins `~/.config/frp/ca.crt` et `~/.frp/ca.crt` lorsqu’ils existent.

**TCP :**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**Hôte virtuel HTTP :**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

Le port local d’OpenChamber reste automatique.

## Commandes utiles

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## Notes de comportement

- un tunnel actif par instance OpenChamber (port)
- démarrer un nouveau mode/fournisseur sur la même instance remplace le tunnel précédent
- générer un nouveau lien de connexion révoque le précédent lien inutilisé
- l’auto-démarrage du tunnel conserve les flags serveur comme `--ui-password` et `--api-only` dans les paramètres d’instance utilisés par les flux de redémarrage/mise à jour
