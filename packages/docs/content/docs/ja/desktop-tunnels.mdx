# デスクトップトンネル

デスクトップアプリでは **Settings → OpenChamber → Tunnel** から公開トンネルを作成できます。この方法では CLI から OpenChamber を起動する必要はありません。

## プロバイダーをインストールする

OpenChamber はあなたのマシン上でプロバイダー CLI を起動します。Cloudflare または Ngrok を使う場合は、使いたいプロバイダーを先にインストールしてください。

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare は `cloudflared` を使います。Ngrok には ngrok アカウントと ngrok ダッシュボードの authtoken が必要です。

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC は手動でインストールする必要がありません。OpenChamber が FRPC 0.70.0 を自動的にダウンロードして管理します。

## アプリから開始する

1. **Settings → OpenChamber → Tunnel** を開きます。
2. **Cloudflare**、**Ngrok**、または **FRPC** を選びます。
3. Cloudflare または Ngrok ではクイックトンネルを開始し、FRPC では設定項目を入力します。
4. 生成された QR コードをスマートフォンでスキャンします。

現在、Ngrok はクイックトンネルに対応しています。Cloudflare はクイックトンネルと管理対象 Cloudflare モードに対応しています。FRPC は自分の FRPS サーバーを使う `managed-remote` の TCP および HTTP 仮想ホストエンドポイントに対応しています。

## FRPC を設定する

FRPC の設定では、プロキシタイプとして **TCP** または **HTTP 仮想ホスト** を選べます。FRPS アドレス、制御ポート、共有トークンを非公開のトークン欄に入力します。CA の入力欄はありません。どちらのタイプでも、OpenChamber がローカルポートを自動設定します。OpenChamber はホスト実行環境のトラストルートからプライベートな一時 PEM を生成して FRPC に注入し、FRPC は引き続き `transport.tls` を有効にして、FRPS アドレスと同じ `serverName` で FRPS の証明書を検証します。OpenChamber が未認証の FRPS から CA を学習したり、証明書検証を無効にしたりすることはありません。FRPS がプライベート CA やエンタープライズ CA を使う場合は、トンネルごとに設定するのではなく、OpenChamber を実行するマシンの OS トラストストアにその CA をインストールするか、`SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS` で公開してください。互換性のため、レガシーなホストパス `~/.config/frp/ca.crt` と `~/.frp/ca.crt` にある CA バンドルも（存在する場合）ホストのトラストルートにマージされます。

- **TCP:** リモートマッピングポートと正確な外部公開 HTTPS オリジンを入力します。FRPC はマッピングされたポートで平文 HTTP を転送するため、その FRPS ポートはループバックだけで待ち受け、指定した URL に Caddy などの TLS 終端を配置してください。OpenChamber は HTTP、不正、またはオリジン以外を含む URL を拒否し、FRPS 制御アドレスからブラウザー TLS を推測しません。
- **HTTP 仮想ホスト:** FRPS ルーティングホスト（custom domain）と外部公開 hostname を入力します。公開 URL は `https://<hostname>` です。Caddy はポート `443` で HTTPS を終端し、FRPS の HTTP 仮想ホストポートへ転送して、`Host` を custom domain、`X-Forwarded-Host` を公開 hostname に設定します。

共有 FRPS トークンは FRPC クライアントを認証しますが、HTTP 仮想ホスト間の分離は提供しません。

## アクセス保護

プロバイダー URL 自体が公開されていても、OpenChamber は独自の接続トークンでアクセスを保護します。生成される接続リンクには一度きりのトークンが含まれ、TTL があり、新しいリンクを生成するかトンネルを停止/再起動すると、未使用の古いリンクは無効化されます。
