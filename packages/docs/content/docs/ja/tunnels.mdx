# トンネル

トンネルは OpenChamber への公開リンクです。別ネットワークの普通のブラウザからアクセスできます。実行中のインスタンスに対して作成するには `openchamber tunnel` を使います。

> **自分のデバイス**（モバイルアプリ、別のデスクトップ）の接続には、通常トンネルは不要です。代わりに [ペアリング](/connect-devices/) して、エンドツーエンド暗号化の [Private Relay](/private-relay/) にセットアップ不要で外出先アクセスを任せましょう。

## 前提条件

OpenChamber はあなたのマシン上でトンネルプロバイダー CLI を起動します。Cloudflare または Ngrok を使う場合は、使いたいプロバイダーを先にインストールしてください。

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare quick tunnel は `cloudflared` で実行できます。Ngrok には ngrok アカウントと ngrok ダッシュボードの authtoken が必要です。

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC は手動でインストールする必要がありません。OpenChamber が FRPC 0.70.0 を自動的にダウンロードして管理します。

## クイックスタート

1. OpenChamber を起動します。

```bash
openchamber
```

この手順を省いた場合、`openchamber tunnel start` は CLI サーバーを自動起動できます。自動起動時には `--port`、`--host`、`--lan`、`--ui-password`、`--api-only` などのサーバーオプションを渡せます。

2. Cloudflare トンネルを開始します。

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

または Ngrok トンネルを開始します。

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. 状態を確認します。

```bash
openchamber tunnel status
```

トンネルが起動すると、`status` に公開 URL が表示されます。それを開くか QR コードをスキャンすると、どこからでも OpenChamber にアクセスできます。

デフォルトでは、OpenChamber は対話型 TTY セッションで QR コードを表示します。QR 出力を強制するには `--qr`、無効にするには `--no-qr` を使います。

## プロバイダー

- `cloudflare`: quick、managed remote、managed local モード
- `ngrok`: quick モード
- `frpc`: 自分で運用する FRPS サーバー経由の TCP または HTTP 仮想ホストエンドポイントを使う managed remote モード

## 管理対象モード

### Managed remote

Cloudflare が管理する token + hostname を使います。

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### Managed local

ローカルの `cloudflared` 設定を使います。

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC（managed-remote）

FRPC は現在の OpenChamber ローカルポートを、自分で運用する FRPS サーバーへ接続します。どちらのエンドポイントでも、OpenChamber がローカル待受ポートを自動的に設定します。

CA の設定は不要です。OpenChamber は実行環境のトラストルートを集めてプライベートな一時 PEM を生成し、FRPC に注入します。FRPC は引き続き `transport.tls` を有効にし、`--frps-address` と同じ `serverName` で FRPS の証明書を検証します。OpenChamber が未認証の FRPS から CA を学習したり、証明書検証を無効にしたりすることはありません。FRPS がプライベート CA やエンタープライズ CA を使う場合は、OpenChamber を実行するマシンの OS トラストストアにその CA をインストールするか、`SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS` で公開してください。トンネルごとの CA 設定はありません。互換性のため、レガシーなホストパス `~/.config/frp/ca.crt` と `~/.frp/ca.crt` にある CA バンドルも（存在する場合）これらのトラストルートにマージされます。Docker では「ホスト」のトラストルートはコンテナのトラストストアなので、コンテナ内に CA をインストールまたはマウントしてください。

#### TCP マッピング

`--remote-port` で TCP マッピングを作成し、`--public-url` で正確な外部 HTTPS オリジンを指定します。

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC は TCP マッピングを通じて平文 HTTP を転送するため、OpenChamber は FRPS アドレスからブラウザー URL を推測しません。`--public-url` には、Caddy などの TLS 終端が待ち受けて FRPS のループバックポートへ転送する `https://` オリジンを指定します。HTTP、認証情報、パス、クエリを含む URL や不正な URL は、接続トークンの発行前に拒否されます。

#### HTTP 仮想ホスト

`--custom-domain <frps-routing-host>` と `--hostname <external-public-host>` を一緒に使います。

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC は `type = "http"` のプロキシを生成します。Caddy は公開ポート `443` で HTTPS を終端し、FRPS の HTTP 仮想ホストポートへ転送します。FRPS がプロキシを選べるよう `Host` を custom domain に設定し、`X-Forwarded-Host` を外部 hostname に設定してください。たとえば FRPS の HTTP 仮想ホストポートが `8080` の場合:

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

公開 URL の形式は `https://<hostname>` です。この例では `https://app.example.com` になります。共有 FRPS トークンは FRPC クライアントを認証しますが、同じトークンを使う他のクライアントからこの HTTP 仮想ホストを分離するものではありません。

`openchamber tunnel` で FRPC トークンを渡せるのは、`--token-file`、`--token-stdin`、または非公開の対話式入力だけです。`--token` は拒否されます。これらの入力で作成したプロファイルは再利用できます。

## プロファイル（managed-remote）

再利用可能なプロファイルを保存します。

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

FRPC TCP では、サーバーとリモートポートの設定も保存します。

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC HTTP 仮想ホストでは次のように保存します。

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

保存したいずれかのプロファイルで起動します。

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## 直接起動用の環境変数

直接起動では、共通の FRPC サーバー設定とトークンを使います。エンドポイントは 1 つだけ選びます。TCP では `OPENCHAMBER_TUNNEL_REMOTE_PORT` と `OPENCHAMBER_TUNNEL_PUBLIC_URL`、HTTP 仮想ホストでは `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN` と `OPENCHAMBER_TUNNEL_HOSTNAME` の両方を使います。FRPS の検証に環境変数は不要です。OpenChamber はホスト実行環境のトラストルート（Docker ではコンテナのトラストストア）と、（存在する場合）`~/.config/frp/ca.crt` と `~/.frp/ca.crt` のレガシー CA バンドルからトラストアンカーを構築します。

**TCP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**HTTP 仮想ホスト:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

OpenChamber のローカルポートは引き続き自動設定されます。

## 便利なコマンド

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## 動作メモ

- OpenChamber インスタンス（ポート）ごとにアクティブなトンネルは 1 つです
- 同じインスタンスで新しい mode/provider を開始すると、前のトンネルは置き換えられます
- 新しい接続リンクを生成すると、以前の未使用リンクは無効化されます
- トンネルの自動起動は、restart/update フローで使うインスタンス設定に `--ui-password` や `--api-only` などのサーバーフラグを保持します
