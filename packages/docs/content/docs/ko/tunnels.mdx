# 터널

터널은 OpenChamber로 연결되는 공개 링크로, 다른 네트워크의 일반 브라우저에서 접근할 수 있게 해줍니다. 실행 중인 인스턴스에 터널을 만들려면 `openchamber tunnel`을 사용하세요.

> **자신의 기기**(모바일 앱, 다른 데스크톱)를 연결할 때는 보통 터널이 필요 없습니다. 대신 [기기를 페어링](/ko/connect-devices/)하고, 설정이 전혀 필요 없는 종단 간 암호화 [Private Relay](/ko/private-relay/)에 집 밖 접속을 맡기세요.

## 사전 요구 사항

OpenChamber는 사용자의 컴퓨터에서 터널 공급자 CLI를 실행합니다. Cloudflare 또는 Ngrok을 사용한다면 먼저 원하는 공급자를 설치하세요:

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare 빠른 터널은 `cloudflared`로 실행할 수 있습니다. Ngrok은 ngrok 계정과 ngrok 대시보드의 authtoken이 필요합니다:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC는 직접 설치할 필요가 없습니다. OpenChamber가 FRPC 0.70.0을 자동으로 다운로드하고 관리합니다.

## 빠른 시작

1. OpenChamber를 시작합니다:

```bash
openchamber
```

이 단계를 건너뛰면 `openchamber tunnel start`가 CLI 서버를 자동으로 시작할 수 있습니다. 자동 시작 시 `--port`, `--host`, `--lan`, `--ui-password`, `--api-only` 같은 서버 옵션을 함께 전달할 수 있습니다.

2. Cloudflare 터널을 시작합니다:

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

또는 Ngrok 터널을 시작합니다:

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. 상태를 확인합니다:

```bash
openchamber tunnel status
```

터널이 올라오면 `status`에 공개 URL이 표시됩니다. 그 URL을 열거나 QR 코드를 스캔하면 어디서든 OpenChamber에 접근할 수 있습니다.

기본적으로 OpenChamber는 대화형 TTY 세션에서 QR 코드를 출력합니다. `--qr`로 QR 출력을 강제하거나 `--no-qr`로 비활성화하세요.

## 공급자

- `cloudflare`: quick, managed remote, managed local 모드
- `ngrok`: quick 모드
- `frpc`: 자체 FRPS 서버를 통해 TCP 또는 HTTP 가상 호스트 엔드포인트를 제공하는 managed remote 모드

## 관리형 모드

### 관리형 원격

Cloudflare가 관리하는 토큰 + 호스트네임을 사용합니다:

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### 관리형 로컬

로컬 `cloudflared` 구성을 사용합니다:

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC (managed-remote)

FRPC는 활성 OpenChamber 로컬 포트를 자체 FRPS 서버에 연결합니다. 두 엔드포인트 유형 모두 OpenChamber가 로컬 수신 포트를 자동으로 제공합니다.

CA 인증서 파일도 필요합니다. FRPS는 SAN이 `--frps-address`와 일치하는 인증서를 제공해야 하며, OpenChamber는 이 신원을 검증하고 인증서 검증을 비활성화하지 않습니다.

#### TCP 매핑

`--remote-port`로 TCP 매핑을 만들고 `--public-url`로 정확한 외부 HTTPS 오리진을 지정하세요:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address 203.0.113.10 --frps-port 7000 --frps-ca-file ~/.config/frp/ca.crt --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC는 TCP 매핑을 통해 암호화되지 않은 HTTP를 전달하므로 OpenChamber는 FRPS 주소에서 브라우저 URL을 추정하지 않습니다. `--public-url`은 Caddy 또는 다른 TLS 종단점이 수신하고 FRPS 루프백 포트로 전달하는 `https://` 오리진이어야 합니다. HTTP, 자격 증명, 경로, 쿼리가 포함된 URL과 잘못된 URL은 연결 토큰을 발급하기 전에 거부됩니다.

#### HTTP 가상 호스트

`--custom-domain <frps-routing-host>`와 `--hostname <external-public-host>`를 함께 사용하세요:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address 203.0.113.10 --frps-port 7000 --frps-ca-file ~/.config/frp/ca.crt --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC는 `type = "http"` 프록시를 생성합니다. Caddy는 공개 포트 `443`에서 HTTPS를 종료하고 FRPS HTTP 가상 호스트 포트로 전달합니다. FRPS가 프록시를 선택하도록 `Host`에는 custom domain을, `X-Forwarded-Host`에는 외부 hostname을 설정해야 합니다. 예를 들어 FRPS HTTP 가상 호스트 포트가 `8080`이라면:

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

공개 URL 형식은 `https://<hostname>`이며, 이 예에서는 `https://app.example.com`입니다. 공유 FRPS 토큰은 FRPC 클라이언트를 인증하지만, 같은 토큰을 사용하는 다른 클라이언트와 이 HTTP 가상 호스트를 격리하지는 않습니다.

`openchamber tunnel`에서 FRPC 토큰은 `--token-file`, `--token-stdin` 또는 비공개 대화형 입력으로만 제공할 수 있으며 `--token`은 거부됩니다. 이 입력 중 하나로 만든 프로필은 다시 사용할 수 있습니다.

## 프로필 (managed-remote)

재사용 가능한 프로필을 저장합니다:

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

FRPC TCP의 경우 서버와 원격 포트 설정도 저장합니다:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address 203.0.113.10 --frps-port 7000 --frps-ca-file ~/.config/frp/ca.crt --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC HTTP 가상 호스트의 경우:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address 203.0.113.10 --frps-port 7000 --frps-ca-file ~/.config/frp/ca.crt --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

저장된 프로필 중 하나로 시작합니다:

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## 직접 시작 환경 변수

직접 시작은 공통 FRPC 서버 설정과 토큰을 사용합니다. 엔드포인트는 하나만 선택하세요. TCP에는 `OPENCHAMBER_TUNNEL_REMOTE_PORT`와 `OPENCHAMBER_TUNNEL_PUBLIC_URL`을 함께 사용하고, HTTP 가상 호스트에는 `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN`과 `OPENCHAMBER_TUNNEL_HOSTNAME`을 함께 사용합니다.

**TCP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: 203.0.113.10
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_TRUSTED_CA_FILE: /home/openchamber/.config/frp/ca.crt
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**HTTP 가상 호스트:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: 203.0.113.10
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_TRUSTED_CA_FILE: /home/openchamber/.config/frp/ca.crt
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

OpenChamber 로컬 포트는 계속 자동으로 제공됩니다.

## 유용한 명령

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## 동작 참고 사항

- OpenChamber 인스턴스(포트)당 활성 터널은 하나입니다
- 같은 인스턴스에서 새 모드/공급자를 시작하면 이전 터널이 대체됩니다
- 새 연결 링크를 생성하면 사용되지 않은 이전 링크가 무효화됩니다
- 터널 자동 시작은 재시작/업데이트 흐름에서 쓰는 인스턴스 설정에 `--ui-password`, `--api-only` 같은 서버 플래그를 저장합니다
