# Tunele w aplikacji desktopowej

Aplikacja desktopowa może utworzyć publiczny tunel z poziomu **Settings → OpenChamber → Tunnel**. W tym trybie nie musisz uruchamiać OpenChamber z CLI.

## Zainstaluj dostawcę

OpenChamber uruchamia CLI dostawcy na Twoim komputerze. W przypadku Cloudflare lub Ngrok najpierw zainstaluj dostawcę, którego chcesz użyć:

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare używa `cloudflared`. Ngrok wymaga konta ngrok i authtokena z panelu ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC nie wymaga ręcznej instalacji. OpenChamber automatycznie pobiera FRPC 0.70.0 i nim zarządza.

## Uruchom z aplikacji

1. Otwórz **Settings → OpenChamber → Tunnel**.
2. Wybierz **Cloudflare**, **Ngrok** albo **FRPC**.
3. Uruchom szybki tunel dla Cloudflare lub Ngrok albo skonfiguruj FRPC.
4. Zeskanuj wygenerowany kod QR telefonem.

Ngrok obsługuje teraz szybkie tunele. Cloudflare obsługuje szybkie tunele i zarządzane tryby Cloudflare. FRPC obsługuje punkty końcowe TCP i wirtualnego hosta HTTP w trybie `managed-remote` przez własny serwer FRPS.

## Skonfiguruj FRPC

Ustawienia FRPC oferują typ proxy **TCP** lub **Wirtualny host HTTP**. Podaj adres FRPS, port sterujący, plik zaufanego certyfikatu CA i wspólny token w prywatnym polu tokenu. Dla obu typów OpenChamber automatycznie podaje lokalny port. FRPS musi przedstawiać certyfikat, którego SAN odpowiada adresowi FRPS; OpenChamber weryfikuje tę tożsamość i nigdy nie wyłącza weryfikacji certyfikatu.

- **TCP:** podaj zdalny port mapowania i dokładne zewnętrzne publiczne źródło HTTPS. FRPC przekazuje nieszyfrowany HTTP przez mapowany port; ogranicz ten port FRPS do interfejsu pętli zwrotnej i umieść Caddy lub inny terminator TLS pod zadeklarowanym adresem. OpenChamber odrzuca adresy HTTP, nieprawidłowe lub zawierające coś poza źródłem i nigdy nie wyprowadza TLS przeglądarki z adresu sterującego FRPS.
- **Wirtualny host HTTP:** podaj host routingu FRPS (custom domain) i zewnętrzny publiczny hostname. Publiczny adres URL to `https://<hostname>`. Caddy kończy HTTPS na porcie `443`, przekazuje ruch do portu HTTP wirtualnych hostów FRPS, ustawia `Host` na custom domain i `X-Forwarded-Host` na publiczny hostname.

Wspólny token FRPS uwierzytelnia klientów FRPC, ale nie zapewnia izolacji między wirtualnymi hostami HTTP.

## Ochrona dostępu

Nawet jeśli URL dostawcy jest publiczny, OpenChamber nadal chroni dostęp własnym connect tokenem. Wygenerowany link zawiera jednorazowy token, ma TTL, a stare nieużyte linki są unieważniane po wygenerowaniu nowego linku albo zatrzymaniu/restarcie tunelu.
