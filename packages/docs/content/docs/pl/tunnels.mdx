# Tunele

Tunel to publiczny link do Twojego OpenChamber, dzięki któremu możesz dotrzeć do niego ze zwykłej przeglądarki w innej sieci. Użyj `openchamber tunnel`, aby utworzyć go dla działającej instancji.

> Łączenie **własnych urządzeń** (aplikacja mobilna, drugi komputer) zwykle nie wymaga tunelu — zamiast tego [sparuj je](/pl/connect-devices/) i pozwól, by szyfrowany end-to-end [Private Relay](/pl/private-relay/) zajął się dostępem spoza domu bez żadnej konfiguracji.

## Wymagania wstępne

OpenChamber uruchamia CLI dostawcy tunelu na Twoim komputerze. W przypadku Cloudflare lub Ngrok najpierw zainstaluj dostawcę, którego chcesz użyć:

```bash
brew install cloudflared
brew install ngrok
```

Szybkie tunele Cloudflare działają przez `cloudflared`. Ngrok wymaga konta ngrok i authtokena z panelu ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC nie wymaga ręcznej instalacji. OpenChamber automatycznie pobiera FRPC 0.70.0 i nim zarządza.

## Szybki start

1. Uruchom OpenChamber:

```bash
openchamber
```

Jeśli pominiesz ten krok, `openchamber tunnel start` może automatycznie uruchomić serwer CLI. Przy auto-starcie możesz przekazać opcje serwera, takie jak `--port`, `--host`, `--lan`, `--ui-password` i `--api-only`.

2. Uruchom tunel Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

Albo uruchom tunel Ngrok:

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. Sprawdź status:

```bash
openchamber tunnel status
```

Gdy tunel działa, `status` pokazuje publiczny adres URL. Otwórz go lub zeskanuj kod QR, aby dotrzeć do OpenChamber z dowolnego miejsca.

Domyślnie OpenChamber wypisuje kod QR w interaktywnych sesjach TTY. Użyj `--qr`, aby wymusić wyświetlenie kodu QR, lub `--no-qr`, aby je wyłączyć.

## Dostawcy

- `cloudflare`: tryby quick, managed remote i managed local
- `ngrok`: tryb quick
- `frpc`: tryb managed remote z punktem końcowym TCP lub wirtualnym hostem HTTP przez własny serwer FRPS

## Tryby zarządzane

### Zarządzany zdalny

Użyj tokenu + nazwy hosta zarządzanych przez Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### Zarządzany lokalny

Użyj lokalnej konfiguracji `cloudflared`:

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC (managed-remote)

FRPC łączy aktywny lokalny port OpenChamber z własnym serwerem FRPS. OpenChamber automatycznie podaje lokalny port nasłuchu dla obu typów punktów końcowych.

Konfiguracja CA nie jest wymagana. OpenChamber zbiera korzenie zaufania środowiska uruchomieniowego, w którym działa, do prywatnego, tymczasowego pliku PEM i wstrzykuje go do FRPC, który nadal włącza `transport.tls` i weryfikuje certyfikat FRPS względem `serverName` równego `--frps-address`. OpenChamber nigdy nie uczy się CA od nieuwierzytelnionego FRPS i nigdy nie wyłącza weryfikacji certyfikatu. Jeśli FRPS używa prywatnego lub firmowego CA, zainstaluj ten CA w magazynie zaufania systemu operacyjnego maszyny, na której działa OpenChamber (albo udostępnij go przez `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`); nie ma już pola CA dla każdego tunelu. Dla zgodności pakiety CA w starszych ścieżkach hosta `~/.config/frp/ca.crt` i `~/.frp/ca.crt` (jeśli istnieją) również są scalane z tymi korzeniami zaufania. W Dockerze „hostowe” korzenie zaufania to magazyn zaufania kontenera — zainstaluj lub zamontuj CA wewnątrz kontenera.

#### Mapowanie TCP

Utwórz mapowanie TCP za pomocą `--remote-port` i podaj dokładne zewnętrzne źródło HTTPS przez `--public-url`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC przekazuje nieszyfrowany HTTP przez mapowanie TCP, dlatego OpenChamber nigdy nie wyprowadza adresu przeglądarki z adresu FRPS. `--public-url` musi być źródłem `https://`, na którym Caddy lub inny terminator TLS nasłuchuje i przekazuje ruch do portu FRPS na interfejsie pętli zwrotnej. Adresy HTTP, z danymi logowania, ścieżką, zapytaniem lub nieprawidłowe są odrzucane przed wydaniem tokenu połączenia.

#### Wirtualny host HTTP

Użyj `--custom-domain <frps-routing-host>` razem z `--hostname <external-public-host>`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC generuje serwer proxy `type = "http"`. Caddy kończy publiczne połączenie HTTPS na porcie `443` i przekazuje je do portu HTTP wirtualnych hostów FRPS. Musi ustawić `Host` na custom domain, aby FRPS wybrał właściwy serwer proxy, oraz `X-Forwarded-Host` na zewnętrzny hostname. Na przykład, gdy port HTTP wirtualnych hostów FRPS to `8080`:

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

Publiczny adres URL ma postać `https://<hostname>`; w tym przykładzie jest to `https://app.example.com`. Wspólny token FRPS uwierzytelnia klientów FRPC, ale nie izoluje tego wirtualnego hosta HTTP od innych klientów używających tego samego tokenu.

Z `openchamber tunnel` FRPC przyjmuje tokeny wyłącznie przez `--token-file`, `--token-stdin` albo prywatny interaktywny monit; `--token` jest odrzucany. Profile utworzone za pomocą jednego z tych źródeł można ponownie wykorzystywać.

## Profile (managed-remote)

Zapisz profil wielokrotnego użytku:

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

Dla FRPC TCP zapisz również ustawienia serwera i zdalnego portu:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

Dla wirtualnego hosta HTTP FRPC:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

Uruchom z jednym z zapisanych profili:

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## Zmienne środowiskowe bezpośredniego uruchomienia

Bezpośrednie uruchomienie używa wspólnych ustawień serwera FRPC i tokenu. Wybierz dokładnie jeden punkt końcowy: `OPENCHAMBER_TUNNEL_REMOTE_PORT` razem z `OPENCHAMBER_TUNNEL_PUBLIC_URL` dla TCP albo razem `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN` i `OPENCHAMBER_TUNNEL_HOSTNAME` dla wirtualnego hosta HTTP. Weryfikacja FRPS nie wymaga żadnej zmiennej środowiskowej: OpenChamber buduje kotwicę zaufania z korzeni zaufania środowiska uruchomieniowego hosta (w Dockerze — magazynu zaufania kontenera), a także ze starszych pakietów CA w `~/.config/frp/ca.crt` i `~/.frp/ca.crt`, jeśli istnieją.

**TCP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**Wirtualny host HTTP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

Lokalny port OpenChamber nadal jest wybierany automatycznie.

## Przydatne polecenia

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## Uwagi dotyczące zachowania

- jeden aktywny tunel na instancję OpenChamber (port)
- uruchomienie nowego trybu/dostawcy na tej samej instancji zastępuje poprzedni tunel
- wygenerowanie nowego linku połączenia unieważnia poprzedni nieużyty
- auto-start tunelu zapisuje flagi serwera, takie jak `--ui-password` i `--api-only`, w ustawieniach instancji używanych przez restarty i aktualizacje
