# Túneis no desktop

O app desktop pode criar um túnel público em **Settings → OpenChamber → Tunnel**. Você não precisa iniciar o OpenChamber pelo CLI para esse fluxo.

## Instale um provedor

O OpenChamber inicia o CLI do provedor na sua máquina. Para Cloudflare ou Ngrok, instale primeiro o provedor que você quer usar:

```bash
brew install cloudflared
brew install ngrok
```

A Cloudflare usa `cloudflared`. O Ngrok exige uma conta ngrok e um authtoken do painel do ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

O FRPC não requer instalação manual. O OpenChamber baixa e gerencia automaticamente o FRPC 0.70.0.

## Inicie pelo app

1. Abra **Settings → OpenChamber → Tunnel**.
2. Escolha **Cloudflare**, **Ngrok** ou **FRPC**.
3. Inicie um túnel rápido com Cloudflare ou Ngrok, ou configure o FRPC.
4. Escaneie o QR code gerado pelo celular.

O Ngrok atualmente suporta túneis rápidos. A Cloudflare suporta túneis rápidos e modos gerenciados da Cloudflare. O FRPC oferece endpoints TCP e de host virtual HTTP no modo `managed-remote` pelo seu servidor FRPS.

## Configure o FRPC

As configurações do FRPC oferecem **TCP** e **Host virtual HTTP** como tipo de proxy. Informe o endereço do FRPS, a porta de controle e o token compartilhado no campo privado de token; não há campo de CA. O OpenChamber fornece automaticamente sua porta local para os dois tipos. O OpenChamber gera um PEM temporário privado a partir das raízes de confiança do ambiente de execução hospedeiro e o injeta no FRPC, que ainda ativa `transport.tls` e verifica o certificado do FRPS com um `serverName` igual ao endereço do FRPS. O OpenChamber nunca aprende uma CA a partir de um FRPS não autenticado e nunca desativa a verificação do certificado. Se o FRPS usa uma CA privada ou corporativa, instale essa CA no repositório de confiança do sistema operacional da máquina que executa o OpenChamber (ou exponha-a por `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`) em vez de configurá-la por túnel. Por compatibilidade, os pacotes de CA nos caminhos legados do host `~/.config/frp/ca.crt` e `~/.frp/ca.crt` (se presentes) também são mesclados às raízes de confiança do host.

- **TCP:** informe a porta remota do mapeamento e a origem HTTPS pública externa exata. O FRPC encaminha HTTP sem criptografia pela porta mapeada; mantenha essa porta FRPS somente no loopback e coloque o Caddy ou outro terminador TLS na URL declarada. O OpenChamber rejeita URLs HTTP, malformadas ou que não sejam apenas uma origem e nunca deduz o TLS do navegador a partir do endereço de controle FRPS.
- **Host virtual HTTP:** informe o host de roteamento do FRPS (custom domain) e o hostname público externo. A URL pública é `https://<hostname>`. O Caddy encerra o HTTPS na porta `443`, encaminha para a porta HTTP de hosts virtuais do FRPS, define `Host` como o custom domain e `X-Forwarded-Host` como o hostname público.

Um token FRPS compartilhado autentica os clientes FRPC, mas não oferece isolamento entre hosts virtuais HTTP.

## Proteção de acesso

Mesmo quando a URL do provedor é pública, o OpenChamber ainda protege o acesso com seu próprio connect token. O link gerado contém um token de uso único, tem TTL, e links antigos não utilizados são revogados quando você gera um novo ou para/reinicia o túnel.
