# Десктопні тунелі

Десктопний застосунок може створити публічний тунель через **Settings → OpenChamber → Tunnel**. Для цього не потрібно запускати OpenChamber через CLI.

## Встановіть провайдера

OpenChamber запускає CLI провайдера на вашій машині. Для Cloudflare або Ngrok спочатку встановіть провайдера, який хочете використовувати:

```bash
brew install cloudflared
brew install ngrok
```

Cloudflare використовує `cloudflared`. Для Ngrok потрібен акаунт ngrok і authtoken з dashboard ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC не потребує ручного встановлення. OpenChamber автоматично завантажує FRPC 0.70.0 і керує ним.

## Запуск із застосунку

1. Відкрийте **Settings → OpenChamber → Tunnel**.
2. Виберіть **Cloudflare**, **Ngrok** або **FRPC**.
3. Запустіть швидкий тунель для Cloudflare чи Ngrok або налаштуйте FRPC.
4. Відскануйте згенерований QR-код з телефона.

Ngrok зараз підтримує швидкі тунелі. Cloudflare підтримує швидкі тунелі й керовані режими Cloudflare. FRPC підтримує кінцеві точки TCP і віртуального хоста HTTP у режимі `managed-remote` через ваш сервер FRPS.

## Налаштування FRPC

У налаштуваннях FRPC можна вибрати тип проксі **TCP** або **Віртуальний хост HTTP**. Введіть адресу FRPS, порт керування та спільний токен у приватному полі токена; поля ЦС немає. Для обох типів OpenChamber підставляє локальний порт автоматично. OpenChamber створює приватний тимчасовий PEM з коренів довіри хостового середовища виконання та впроваджує його в FRPC, який і далі вмикає `transport.tls` і перевіряє сертифікат FRPS за `serverName`, що дорівнює адресі FRPS. OpenChamber ніколи не дізнається ЦС від неавтентифікованого FRPS і ніколи не вимикає перевірку сертифіката. Якщо FRPS використовує приватний або корпоративний ЦС, встановіть цей ЦС у сховище довіри ОС машини, на якій працює OpenChamber (або відкрийте його через `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`), а не налаштовуйте його для кожного тунелю. Для сумісності набори ЦС за застарілими шляхами хоста `~/.config/frp/ca.crt` та `~/.frp/ca.crt` (якщо вони існують) також об'єднуються з коренями довіри хоста.

- **TCP:** введіть віддалений порт зіставлення й точне зовнішнє публічне джерело HTTPS. FRPC пересилає незашифрований HTTP через зіставлений порт; залиште цей порт FRPS доступним лише через інтерфейс зворотного зв’язку та розмістіть Caddy або інший термінатор TLS за вказаною URL-адресою. OpenChamber відхиляє URL-адреси HTTP, неправильного формату або такі, що не є лише джерелом, і ніколи не визначає TLS браузера з адреси керування FRPS.
- **Віртуальний хост HTTP:** введіть маршрутний хост FRPS (custom domain) і зовнішній публічний hostname. Публічна URL-адреса: `https://<hostname>`. Caddy завершує HTTPS на порту `443`, пересилає запити на HTTP-порт віртуальних хостів FRPS, задає `Host` як custom domain, а `X-Forwarded-Host` — як публічний hostname.

Спільний токен FRPS автентифікує клієнти FRPC, але не забезпечує ізоляцію між віртуальними хостами HTTP.

## Захист доступу

Навіть якщо URL провайдера публічний, OpenChamber все одно захищає доступ власним connect token. Згенероване посилання містить одноразовий token, має TTL, а старі невикористані посилання відкликаються, коли ви генеруєте нове або зупиняєте/перезапускаєте тунель.
