# Тунелі

Тунель — це публічне посилання на ваш OpenChamber, щоб дістатися до нього зі звичайного браузера в іншій мережі. Скористайтеся `openchamber tunnel`, щоб створити його для запущеного інстансу.

> Для підключення **власних пристроїв** (мобільного застосунку, іншого десктопа) тунель зазвичай не потрібен — краще [зв'яжіть їх](/uk/connect-devices/) і дозвольте наскрізно зашифрованому [Private Relay](/uk/private-relay/) забезпечити доступ поза домом без жодних налаштувань.

## Передумови

OpenChamber запускає CLI провайдера тунелю на вашій машині. Для Cloudflare або Ngrok спочатку встановіть провайдера, який хочете використовувати:

```bash
brew install cloudflared
brew install ngrok
```

Швидкі тунелі Cloudflare працюють через `cloudflared`. Для Ngrok потрібен акаунт ngrok і authtoken з dashboard ngrok:

```bash
ngrok config add-authtoken <your-ngrok-token>
```

FRPC не потребує ручного встановлення. OpenChamber автоматично завантажує FRPC 0.70.0 і керує ним.

## Швидкий старт

1. Запустіть OpenChamber:

```bash
openchamber
```

Якщо пропустити цей крок, `openchamber tunnel start` може автоматично запустити CLI server. Під час auto-start можна передати server options: `--port`, `--host`, `--lan`, `--ui-password` і `--api-only`.

2. Запустіть тунель Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode quick
```

Або запустіть тунель Ngrok:

```bash
openchamber tunnel start --provider ngrok --mode quick
```

3. Перевірте статус:

```bash
openchamber tunnel status
```

Коли тунель піднято, `status` показує публічний URL. Відкрийте його або відскануйте QR-код, щоб дістатися до OpenChamber звідусіль.

Типово OpenChamber виводить QR-код в інтерактивних TTY-сесіях. Використовуйте `--qr`, щоб примусово вивести QR, або `--no-qr`, щоб вимкнути його.

## Провайдери

- `cloudflare`: quick, managed remote і managed local режими
- `ngrok`: quick режим
- `frpc`: режим managed remote із TCP або віртуальним хостом HTTP через власний сервер FRPS

## Керовані режими

### Кероване віддалене

Використовуйте токен + хостнейм, керовані Cloudflare:

```bash
openchamber tunnel start --provider cloudflare --mode managed-remote --token-file ~/.secrets/cf-token --hostname app.example.com
```

### Кероване локальне

Використовуйте локальний конфіг `cloudflared`:

```bash
openchamber tunnel start --provider cloudflare --mode managed-local --config ~/.cloudflared/config.yml
```

### FRPC (managed-remote)

FRPC з’єднує активний локальний порт OpenChamber із вашим сервером FRPS. Для обох типів кінцевих точок OpenChamber підставляє локальний порт автоматично.

Налаштування ЦС не потрібне. OpenChamber збирає корені довіри середовища виконання, в якому працює, у приватний тимчасовий PEM і впроваджує його в FRPC, який і далі вмикає `transport.tls` та перевіряє сертифікат FRPS за `serverName`, що дорівнює `--frps-address`. OpenChamber ніколи не дізнається ЦС від неавтентифікованого FRPS і ніколи не вимикає перевірку сертифіката. Якщо FRPS використовує приватний або корпоративний ЦС, встановіть цей ЦС у сховище довіри ОС машини, на якій працює OpenChamber (або відкрийте його через `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS`); поля ЦС для кожного тунелю більше немає. Для сумісності набори ЦС за застарілими шляхами хоста `~/.config/frp/ca.crt` та `~/.frp/ca.crt` (якщо вони існують) також об'єднуються з цими коренями довіри. У Docker «хостові» корені довіри — це сховище довіри контейнера, тому встановіть або змонтуйте ЦС всередині контейнера.

#### Зіставлення TCP

Створіть зіставлення TCP за допомогою `--remote-port` і вкажіть точне зовнішнє джерело HTTPS через `--public-url`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

FRPC пересилає незашифрований HTTP через зіставлення TCP, тому OpenChamber ніколи не визначає URL-адресу браузера з адреси FRPS. `--public-url` має бути джерелом `https://`, на якому Caddy або інший термінатор TLS приймає з’єднання та пересилає їх на локальний порт FRPS. URL-адреси HTTP, з обліковими даними, шляхом, запитом або неправильного формату відхиляються до видачі токена підключення.

#### Віртуальний хост HTTP

Використовуйте `--custom-domain <frps-routing-host>` разом із `--hostname <external-public-host>`:

```bash
openchamber tunnel start --provider frpc --mode managed-remote --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

FRPC створює проксі `type = "http"`. Caddy завершує публічний HTTPS на порту `443` і пересилає запити на HTTP-порт віртуальних хостів FRPS. Заголовок `Host` має містити custom domain, щоб FRPS вибрав потрібний проксі, а `X-Forwarded-Host` — зовнішній hostname. Наприклад, якщо HTTP-порт віртуальних хостів FRPS — `8080`:

```caddyfile
app.example.com {
  reverse_proxy 127.0.0.1:8080 {
    header_up Host openchamber.internal
    header_up X-Forwarded-Host app.example.com
  }
}
```

Публічна URL-адреса має формат `https://<hostname>`; у цьому прикладі це `https://app.example.com`. Спільний токен FRPS автентифікує клієнти FRPC, але не ізолює цей віртуальний хост HTTP від інших клієнтів із тим самим токеном.

Для `openchamber tunnel` FRPC приймає токен лише через `--token-file`, `--token-stdin` або приватний інтерактивний запит; `--token` заборонено. Профілі, створені одним із цих способів, можна використовувати повторно.

## Профілі (managed-remote)

Збережіть багаторазовий профіль:

```bash
openchamber tunnel profile add --provider cloudflare --mode managed-remote --name prod-main --hostname app.example.com --token-file ~/.secrets/cf-token
```

Для FRPC TCP збережіть налаштування сервера та віддаленого порту:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-tcp --frps-address frps.example.com --frps-port 7000 --remote-port 20000 --public-url https://app.example.com:20000 --token-file ~/.secrets/frp-token
```

Для віртуального хоста HTTP FRPC:

```bash
openchamber tunnel profile add --provider frpc --mode managed-remote --name frpc-http --frps-address frps.example.com --frps-port 7000 --custom-domain openchamber.internal --hostname app.example.com --token-file ~/.secrets/frp-token
```

Запустіть з одним зі збережених профілів:

```bash
openchamber tunnel start --profile prod-main
openchamber tunnel start --profile frpc-tcp
openchamber tunnel start --profile frpc-http
```

## Змінні середовища для прямого запуску

Для прямого запуску потрібні спільні налаштування сервера FRPC і токен. Виберіть рівно одну кінцеву точку: `OPENCHAMBER_TUNNEL_REMOTE_PORT` разом з `OPENCHAMBER_TUNNEL_PUBLIC_URL` для TCP або разом `OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN` і `OPENCHAMBER_TUNNEL_HOSTNAME` для віртуального хоста HTTP. Для перевірки FRPS не потрібна жодна змінна середовища: OpenChamber будує якір довіри з коренів довіри хостового середовища виконання (у Docker — сховища довіри контейнера), а також із застарілих наборів ЦС за шляхами `~/.config/frp/ca.crt` та `~/.frp/ca.crt`, якщо вони існують.

**TCP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_REMOTE_PORT: 20000
OPENCHAMBER_TUNNEL_PUBLIC_URL: https://app.example.com:20000
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

**Віртуальний хост HTTP:**

```yaml
OPENCHAMBER_TUNNEL_PROVIDER: frpc
OPENCHAMBER_TUNNEL_MODE: managed-remote
OPENCHAMBER_TUNNEL_SERVER_ADDRESS: frps.example.com
OPENCHAMBER_TUNNEL_SERVER_PORT: 7000
OPENCHAMBER_TUNNEL_CUSTOM_DOMAIN: openchamber.internal
OPENCHAMBER_TUNNEL_HOSTNAME: app.example.com
OPENCHAMBER_TUNNEL_TOKEN: <token>
```

Локальний порт OpenChamber і надалі визначається автоматично.

## Корисні команди

```bash
openchamber tunnel providers
openchamber tunnel ready --provider cloudflare
openchamber tunnel ready --provider ngrok
openchamber tunnel ready --provider frpc
openchamber tunnel doctor --provider cloudflare
openchamber tunnel doctor --provider ngrok
openchamber tunnel doctor --provider frpc
openchamber tunnel stop --port 3000
```

## Нотатки щодо поведінки

- один активний тунель на інстанс OpenChamber (порт)
- запуск нового режиму/провайдера на тому самому інстансі замінює попередній тунель
- генерація нового посилання для підключення відкликає попереднє невикористане
- авто-старт тунелю зберігає серверні прапорці на кшталт `--ui-password` та `--api-only` у налаштуваннях інстансу, які використовуються під час рестарту/оновлення
